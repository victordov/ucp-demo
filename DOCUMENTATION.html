Agentic Commerce, end to end

A complete working implementation of the Universal Commerce Protocol (UCP 2026-04-08) and the Agent Payments Protocol (AP2): an AI shopping agent that discovers merchants, negotiates capabilities, builds carts, and pays — with every step cryptographically signed, verified, and auditable.

4 microservicesJSON-RPC 2.0 / MCP PKI · no DIDsRFC 9421 signatures AP2 mandates · SD-JWT+kb51 e2e + 20 schema + 28 conformance15 demo scenarios

1What is this project?

Read this first if you're new — no protocol knowledge required.

Imagine telling an AI assistant: “Find me noise-cancelling headphones under $300, delivered in 2 days” — and the assistant searches several stores, compares live prices, fills a cart, and pays, all by itself. That raises two hard questions:

🛍 How do machines shop?

Web shops are built for humans clicking buttons. Agents need a machine-readable, standard way to search catalogs, create checkouts and place orders at any store without custom integrations. That standard is UCP.

🔐 How do we trust an agent with money?

If an AI pays on your behalf, the merchant and the bank need cryptographic proof that you authorized exactly this purchase — not the agent's hallucination of it. That proof system is AP2 and its signed mandates.

This repository implements both, as four small services you can run with one command (npm run dev) and watch interacting in real time. The chat UI at localhost:4100 has a Protocol trace panel — every event in it is a real signed request on the wire, not a simulation.

How to read this page. Sections are ordered from beginner to advanced — the colored badges in the sidebar (intro core deep) show the depth. You can stop after section 4 and still understand what the system does.

2The four services

Each service plays one role from the UCP/AP2 ecosystem. They run as independent processes and only talk to each other over signed JSON-RPC — exactly as separate companies would on the open internet. Click a card for its purpose, internal architecture and implementation details.

🤖 Shopping Agent :4100

“Shoppy” — the UCP Platform and AP2 Shopping Agent. Hosts the chat UI (manual, LLM chat, autonomous agent, 24-flow scenario console), discovers merchants, orchestrates the whole purchase and renders the live protocol trace.

Click for details ▾
🏪 Merchant Portal :4101

Hosts four UCP Businesses (Wavelength, SoundHub, ElectroMart, AudioNest) with the integrated Merchant Agent: catalog, cart, checkout lifecycle, AP2 verification, orders, refunds/disputes, OAuth identity linking, signed webhooks and an operable dashboard.

Click for details ▾
👛 Credentials Provider :4102

“Walletly” — the AP2 Credentials Provider and the user's consent surface: card vault (PCI scope), user device key, SD-JWT+kb issuance, single-use tokens, spend policy, approval inbox, passkeys.

Click for details ▾
🏦 Payment Provider :4103

“PayStream” — the PSP / AP2 Merchant Payment Processor plus the network trust layer: independent mandate re-verification, multi-rail authorize/capture/refund, KYA agent registry, reputation, velocity rules.

Click for details ▾
Why four? Separation of concerns is the security model: the agent never sees card numbers, the merchant never sees the wallet, and the PSP trusts mathematics — not the agent's word.

The AP2 role-based ecosystem (overview §3)

AP2 defines six roles with a strict separation of concerns. This demo realizes them across four services — the Trusted Surface and the Network & Issuer are combined into other components, which AP2 explicitly permits (non-normative role combinations).

AP2 role (§3)Played here byResponsibility
Shopping Agent (SA)🤖 Shopping Agent · “Shoppy” (:4100)Product discovery, builds the Checkout, executes the purchase.
Merchant (M)🏪 Merchant Portal · 4 tenants (:4101)Owns the catalog, creates and signs the Checkout (merchant_authorization), fulfills orders.
Credential Provider (CP)👛 Credentials Provider · “Walletly” (:4102)Source of payment credentials; verifies the agent is authorized to use a credential and scopes it (single-use token).
Merchant Payment Processor (MPP)🏦 Payment Provider · “PayStream” (:4103)Processes the payment; verifies the credential was authorized to pay for this Checkout.
Trusted Surface (TS) combinedHosted by the CP (biometric / Touch ID consent) & the PSP (3-DS step-up page)Gets informed user consent for the intent before a user-signed Mandate is created. AP2 allows the CP/agent to host it — here the CP does (Trusted Platform Provider model).
Network & Issuer simulatedStood in for by the PSP + the chosen test cardPayment network + issuer of the user's credential; authorizes transactions and issues agent-transaction tokens. Simulated deterministically in this demo.

Per AP2's non-normative examples, roles may be combined — e.g. an SA may provide its own CP, a Merchant may be its own MPP, and the CP may host the Trusted Surface. This demo keeps SA / Merchant / CP / MPP as separate processes (the hardest, most honest case) and folds TS + Network/Issuer in.

Reading the diagram — compliance per edge

UCP-compliantAgent → Merchant capability calls (indigo) and the order webhook (teal): the shapes are the official UCP JSON Schemas, the transport is the UCP MCP binding (JSON-RPC 2.0 tools/call with meta["ucp-agent"] identity, dual structuredContent+content[] output, spec error codes), discovery is /.well-known/ucp. AP2-compliantThe violet and amber edges implement the AP2 roles: wallet operations and mandate signing at the Credentials Provider, processing at the PSP, credential release between them. AP2 doesn't prescribe a transport — we reuse the same signed MCP binding so one trust fabric covers everything. REST (no signing)Profile discovery (per spec), OAuth 2.0 Identity Linking (RFC 8414 + PKCE), the user's consent surface (Walletly) and the 3-DS challenge page. These are browser/HTTP conventions, not UCP capability calls. not implementedUCP's REST/OpenAPI, A2A (Agent Card) and Embedded bindings — the tool handlers are transport-agnostic, so these are routing additions (see Proposition → roadmap).

Three ways to drive the same protocol

The flow above is identical no matter how it's triggered — only who does the choosing changes. The Shoppy UI makes the mode explicit:

🛍 Manual

You type a request, then you pick the merchant, shipping option and payment method — interactive product cards → checkout card → Google-Pay sheet → receipt.

🤖 LLM chat / autonomous

With an API key, a real LLM drives the chat turn by turn (/api/chat) — it searches, presents options and asks before paying. Or let it run the whole purchase from one goal (/api/agent). Toggle 📜 Scripted ⟷ 🤖 LLM by the composer.

⚙️ Scenarios

One-click automated demos (15 of them) for success, failure and capability flows — the fastest way to watch a specific protocol path in the trace. See §10.

📜 Scripts

npm run e2e / validate / conformance drive the same flows headlessly for testing. npm run stop shuts the servers down.

2 · Identity Linking (dev.ucp.common.identity_linking) — OAuth 2.0

Purpose: until now the agent is an anonymous (if verified) buyer — the merchant sees a KYA-verified agent, not your customer. Identity Linking lets the user connect their existing merchant account to the agent via standard OAuth 2.0, so the agent can act on user-scoped data: read order history ("where is my package?", reorders), returns, loyalty and personalization — with access bounded by explicit scopes (dev.ucp.shopping.order:read in our build) instead of a password. The token only unlocks user-scoped tools; all purchasing still requires the full AP2 mandate chain — a leaked token cannot spend money. Scenario: identity.

6Discovery & /.well-known/ucp

Before two parties can transact they must find each other and agree on what they both speak. UCP does this with a single, cacheable JSON document: the profile.

One document, two jobs

A profile simultaneously declares capabilities (for negotiation) and publishes signing keys (for identity). This enables permissionless onboarding: any platform with a fetchable profile can interact with any business — no API-key exchange required.

ucp.versionProtocol version (2026-04-08) — date-based, negotiated per session. ucp.servicesWhere to call: transport (mcp here) + the endpoint URL. Platforms resolve endpoints from this — never hardcoded. ucp.capabilitiesWhat it can do — reverse-domain names (dev.ucp.shopping.checkout), each with required spec + schema URLs, and extends for extensions. ucp.payment_handlersHow to pay — handler registry (com.google.pay) with instance id and config. signing_keysPublic ES256 keys as JWKs — the PKI roots for everything in §7–8.

Who exposes a well-known endpoint? Every business MUST (here: one per merchant tenant). The platform publishes one so businesses can verify it from the UCP-Agent header. 7Communication: JSON-RPC 2.0 & the MCP binding

All service-to-service communication uses JSON-RPC 2.0 over HTTP in the exact shape UCP specifies for its MCP transport: a single tools/call method wrapping named operations.

The envelope

JSON-RPC 2.0 is a tiny convention: a request has method, params, id; a response has a matching id and either result or error. UCP profiles it further:

  • Operation name goes in params.name (e.g. create_checkout), payload in params.arguments.
  • arguments.meta["ucp-agent"].profile is required on every call — it carries the caller's identity for negotiation; meta["idempotency-key"] is required for complete_checkout / cancel_checkout.
  • Responses use the dual-output pattern: structured payload in result.structuredContent plus a serialized copy in result.content[] for older MCP clients.
  • The MCP handshake methods initialize and tools/list are implemented; tool definitions declare outputSchema referencing official UCP JSON Schemas.

8UCP in depth

The Universal Commerce Protocol decomposes commerce into capabilities with strict JSON Schemas, composable extensions, and pluggable payment handlers.

Checkout lifecycle

The merchant recomputes totals and re-signs the checkout on every transition. Totals follow the schema rules: exactly one subtotal and one total, discounts negative, shipping as type fulfillment.

Payment handler model

UCP separates what is accepted (instruments) from how it's processed (handlers). The credential flows one way — platform → business → PSP — and the agent only ever holds opaque tokens:

1 · NegotiationMerchant advertises com.google.pay (instance gpay_wavelength) in profile + checkout responses. 2 · AcquisitionAgent executes the handler against the wallet: mint_instrument → single-use network token + cryptogram. The PAN never appears. 3 · CompletionAgent submits the instrument with handler_id; merchant validates it against the advertised set, forwards to the PSP.

For agents, the credential is an AP2 composite token: { network_token, payment_mandate } — token and proof travel together (§8).

Capability negotiation (the full handshake)

Per the spec's Negotiation Protocol, platform and business agree on a capability set per session:

  1. Profile advertisement — every request carries meta["ucp-agent"].profile + the UCP-Agent header.
  2. Discovery — the platform fetches each business /.well-known/ucp (cached with a 60s TTL floor, per the spec, regardless of Cache-Control).
  3. Namespace validation — each capability's spec & schema URL origin MUST match its reverse-domain authority (dev.ucp.* → https://ucp.dev); offending capabilities are rejected before negotiation.
  4. Intersection — name match → highest mutual version → transitive orphan-extension pruning (single- and multi-parent). dev.ucp.shopping.ap2_mandate is always in the result, so every session is security-locked.
  5. Schema resolution — fetch base + active extension schemas, compose via allOf, validate payloads (§13).

The business side recomputes the same intersection on each request: the merchant validates the platform profile's shape + version and exposes the active set; an empty intersection → capabilities_incompatible (a 200 business outcome with a continue_url). Inter-service calls (Merchant→PSP, PSP→CP) are peer calls, so negotiation is enforced only on the platform-facing merchant endpoint. The agent's Protocol Trace shows the Namespace Validation, Capability Negotiation and Schema Resolution steps live.

Amounts, addresses and other schema conventions (click to expand)
  • Amounts are minor units (integer cents) everywhere on the wire: $274.00 → 27400. The UIs divide by 100 only for display.
  • Addresses use schema.org-style fields: street_address, address_locality, address_region, postal_code, address_country.
  • Catalog products follow product.json: title, structured description {plain|html|markdown}, price_range, variants[] with price/list_price, plus tags/metadata for vendor extras.
  • Orders follow order.json: line items with object quantities {original, total, fulfilled} and a status, fulfillment expectations (what will happen) vs append-only events (what did happen).
  • Names are governed by reverse-domain namespaces: dev.ucp.* is reserved for the UCP body; vendors must use their own (org.ap2.* for our AP2 services). Spec/schema URL origins must match the namespace authority.

9Security I — PKI & HTTP message signatures

Identity in this system is a URL plus a key pair — no DIDs, no central registry. Every request and webhook is signed with RFC 9421 HTTP Message Signatures and verified before any business logic runs.

What signatures defeat

AttackDefense
ImpersonationKey resolved from the claimed profile — you can't claim an identity whose private key you don't hold.
TamperingContent-Digest binds the exact body bytes; any change breaks the signature.
Cross-host replay@authority + @path are signed — a request for one host can't be replayed at another.
Same-host replaySigned Idempotency-Key: duplicates return the cached response; same key + different body → 409.
Identity confusionmeta["ucp-agent"] must equal the signed UCP-Agent header — verified binding.

Trust model

Two layers, per AP2's short-term model:

  1. Authenticity (cryptographic): the signature proves who sent it and what was sent.
  2. Authorization (policy): each service holds a curated allowlist of peer profile URLs. Authentic-but-unknown callers get 403 profile_not_trusted before any profile fetch — capping the discovery budget against malicious URLs.

Keys are ES256 (ECDSA P-256/SHA-256) with raw r‖s encoding — mind the DER/IEEE-P1363 trap if you port this. Webhooks must be signed; the agent rejects unsigned order events. Demo simplification: keys are generated in-memory at boot and transport is plain localhost HTTP — the verification pipeline is unchanged from production behavior.

10Security II — AP2 & the mandate chain

HTTP signatures authenticate services. Mandates authenticate the human. Per AP2's principle of verifiable intent, not inferred action (overview §2.3), trust is never based on a language model's probabilistic output — it is anchored to deterministic, non-repudiable proof of intent from all parties: the user signs the Checkout + Payment Mandates, the merchant signs the Checkout. These credentials prove — durably, for disputes months later — that the user authorized exactly this purchase, and defend against agent “hallucination” or misinterpretation.

Two core journeys (overview §4)

§4.1 Human Present direct

The user is present to approve the closed Checkout + Payment Mandates directly.

  1. Setup — the user connects the agent to a Credential Provider.
  2. Discovery & negotiation — the agent assembles a cart across merchants.
  3. Merchant validates cart — the merchant signs the Checkout it will fulfill.
  4. Provide payment methods — the agent selects an instrument option from the CP.
  5. Show cart — the agent presents the final cart on a Trusted Surface.
  6. Sign & pay — the user's biometric approval creates the signed Checkout Mandate (+ Payment Mandate, linked by the checkout hash).
  7. Payment execution → issuer → (challenge) → authorize — Checkout + Payment receipts returned.
§4.2 Human Not Present autonomous

The user approves open mandates up front; the agent then acts alone, presenting agent-signed closed mandates.

  1. Phase 1a (present) — the agent repeats the task back; the user authenticates (biometric) and signs the Open Checkout + Payment Mandates (the conditions), each carrying the agent key as cnf. The user leaves.
  2. Phase 1b (not present) — the agent shops alone and opens a merchant-signed Checkout that meets the conditions.
  3. Phase 2 (not present) — the agent signs the Closed Checkout + Payment Mandates with its own key; sd_hash binds each closed mandate to its open one. Merchant + PSP verify the closed mandates satisfy the open constraints.

Fallback: the Merchant (or CP) may return an unresolved_constraint error to pull the user back in-session — turning a human-not-present flow into a human-present one.

Key discovery is uniform: all public keys are JWKs in profile signing_keys. The user device key is published in the CP profile — that's how the merchant and PSP resolve it.

Why JCS canonicalization?

HTTP signatures hash raw bytes — fine for transient requests verified on the same connection. Mandates are different: they're durable evidence, stored for months and re-serialized by every system they pass through (merchant → PSP → card network). RFC 8785 (JCS) guarantees that semantically identical JSON always produces byte-identical input, so the signature is reproducible by anyone, anytime.

The merchant's rule: sign JCS(checkout − ap2) as a detached JWS (header..signature, RFC 7515 App. F) — excluding the ap2 field makes the scheme self-referential-safe and forward-compatible with future AP2 fields.

The checkout mandate is a real SD-JWT+kb

ap2.checkout_mandate isn't a plain JWS — it's a proper SD-JWT with Key Binding (dc+sd-jwt~kb), the verifiable-credential format AP2 calls for:

issuerThe Credentials Provider signs the credential with its service key (it issued the user's device key at enrollment). holder (key binding)A Key-Binding JWT (typ=kb+jwt) signed by the user's device key binds the presentation to aud = this merchant's profile, nonce = the checkout id, and sd_hash — so the same credential can't be replayed at another merchant or for another checkout. selective disclosureClaims like sub (user id) and buyer_email are split into disclosures; the agent reveals the user id but withholds the email — the merchant verifies the credential without learning data it doesn't need. nested bindingThe credential embeds the full merchant-signed checkout, so the user's key-binding signature transitively covers the merchant's signature over the exact terms.

Wire format: <issuer-jwt>~<disclosure>…~<kb-jwt>. The e2e suite validates this structure (issuer JWT + disclosures + a kb+jwt binding aud/nonce/sd_hash).

The payment mandate is an SD-JWT+kb too (mandate.payment.1)

The payment_mandate inside the composite token is a proper AP2 verifiable credential — issued by the Credentials Provider, key-bound to the user device key, carrying the canonical AP2 fields:

vctmandate.payment.1 — the verifiable-credential type, asserted exactly by the PSP. transaction_idbase64url SHA-256 of the merchant-signed checkout (JCS(checkout − ap2)). The merchant forwards the same hash and the PSP asserts transaction_id == checkout_hash — so the payment is bound to the exact signed terms, not just a session id. payment_amount / payee / payment_instrumentAP2 Amount (minor units), Merchant object, and PaymentInstrument (the instrument is selectively disclosable). key bindingaud = the merchant profile (payee), nonce = the checkout id → can't be replayed at another merchant or checkout.

The PSP verifies it with the same SD-JWT+kb machinery as the checkout mandate (sdJwtVerify), then re-checks vct, transaction_id, amount, payee and expiry before pulling credentials.

Signed receipts close the loop

On success each verifier returns a signed AP2 receipt (compact JWS, typ=ap2-receipt+jwt), so both sides hold non-repudiable proof of the outcome:

Checkout ReceiptIssued by the merchant at complete_checkout: {status, iss, iat, reference, order_id}, where reference = hash of the closed checkout mandate. Payment ReceiptIssued by the PSP on authorize: {status, iss, iat, reference, payment_id, psp_confirmation_id, network_confirmation_id}, where reference = hash of the closed payment mandate.

Both ride back on the completed checkout's ap2 object (next to merchant_authorization), are stored in the order's evidence bundle, and surface in the agent's Protocol Trace as a dedicated AP2 Receipts card.

Agent-economy controls — spend policy, KYA, multi-rail, audit

When agents become economic actors, the protocol layer alone isn't enough: the user needs to constrain what the agent may spend, the network needs to know and score the agent, payments need the right rail per transaction, and everything must leave a tamper-evident record. This suite implements all four pillars (inspired by the payment networks' agentic-commerce frameworks — agentic tokens, Know-Your-Agent, multi-rail settlement).

1 · Spend controls & programmable payment Walletly → Agent permissions

A per-agent consent policy lives at the Credentials Provider — the user's trusted surface — and works like a network-enforced agentic-token scope:

autonomyalways_ask · ask_above (autonomous below a threshold) · autonomous — gates human-not-present purchases capsper-transaction cap + total budget with a live spent meter (minor units) allowlistoptional merchant allowlist + validity window + preferred settlement rail enforced 3×mint_instrument (the policy snapshot is bound into the single-use token) → sign_mandate(PaymentMandate) → PSP re-check of the token-bound scope at authorization (defense in depth) errorspolicy_per_tx_cap_exceeded · policy_budget_exceeded · policy_merchant_not_allowed · policy_autonomy_violation · policy_window_expired

Scenarios: over_budget, merchant_blocked, autonomy.

2 · Verified authorization across parties

The user's authorization no longer stops at the agent. At complete_checkout the agent presents the user-signed Checkout Mandate (ap2.checkout_mandate, SD-JWT+kb); the merchant verifies the signature + key-binding against the CP's published JWKs and confirms the embedded terms match the live session, then the Payment Mandate travels to the PSP, which re-validates independently (signature, transaction_id == checkout hash, amount, payee). Both verification logs record the mandate — the purchase is provably what the user authorized, at every hop. For autonomous (human-not-present) purchases, the user-signed Open Checkout/Payment Mandates carry the constraints (allowed merchants, amount range, budget) and each closed mandate is verified to satisfy them — exactly the bound the agent must shop within.

3 · Multi-rail settlement com.google.pay + com.paystream.rtp

Merchants and the wallet advertise two UCP payment handlers; the agent selects the optimal rail per transaction and the choice is visible in the trace ("Rail selection"):

card networksingle-use network token via Google Pay — wide acceptance, chargeback rights, auth → capture RTPinstant bank transfer from an RTP-capable checking account — instant fiat settlement, no card network selectionexplicit choice > the policy's preferred_rail > auto (≥ $500 → RTP, else card)

Scenario: rtp_rail. The PSP records the rail on every transaction.

4 · Know Your Agent (KYA) + reputation PayStream registry

The PSP hosts an agent registry — KYC for agents. Each agent has a KYA level (verified/basic), a status, and a behavior-driven reputation score (clean capture +1, decline −8, policy violation −8, velocity −10). The merchant calls lookup_agent before completing a checkout and refuses suspended or low-reputation agents (agent_untrusted); the PSP enforces the same gate using the agent claim inside the PaymentMandate. Scenario: kya_blocked.

5 · Velocity rules — agent-native fraud detection

Agent traffic is thousands of transactions per second, not dozens per day. The PSP rate-limits authorizations per agent over a rolling 60-second window (PSP_VELOCITY_LIMIT, default 20). Exceeding it declines with velocity_exceeded (HTTP 429) and dings the agent's reputation. Scenario: velocity (limit lowered to 2, third rapid purchase declined).

6 · Immutable audit trail + evidence bundle

Every trace event is hash-chained: hash = SHA-256(seq, prev_hash, core fields, payload hash). Editing any past event breaks every subsequent hash — a tamper-evident log. POST /api/audit/verify re-derives the chain; Export evidence bundle (chip after any purchase) downloads a dispute-ready JSON with all signed mandates (the Checkout + Payment Mandates as SD-JWT+kb; open mandates when autonomous) plus the chained event log.

7 · Post-purchase agency — delivery monitoring

The agent's job doesn't end at payment. Check delivery promise re-reads the order and compares the carrier's current expectation (fulfillment.expectations[0].fulfillable_on) with the promise made at checkout. If the ETA slipped, the agent proactively secures a 10% partial refund through the standard refund flow (PSP credit + signed order adjustment + webhook). Scenario: late_delivery.

12Observability — logs & request tracing

A protocol you can't watch is a protocol you can't trust. Every service keeps an audit trail, and the agent streams a live wire-level trace into the UI.

The Protocol Trace (agent UI)

The orchestrator wraps every outbound tools/call and emits structured TraceEvents over Server-Sent Events (GET /api/trace/{session}) to the “Protocol trace” panel:

  • UCP request/response pairs — full JSON-RPC body plus the actual signature headers (Signature-Input, Content-Digest, Idempotency-Key) and round-trip latency,
  • AP2 mandate cards — payload, seal type, truncated JWS,
  • PKI verification events — e.g. “verify merchant_authorization: kid=wavelength-2026 ✓”.

Events are buffered per session, so reconnecting replays the full history.

Correlating a request across services

Several identifiers tie the distributed story together:

Idempotency-KeyUnique per call, signed, logged by caller and callee — the per-request correlation id. checkout_idBinds agent session ↔ merchant session ↔ mandates ↔ PSP transaction ↔ order ↔ webhook. mandate idschk_… → pay_… (the Checkout + Payment Mandates) — the authorization chain, visible in the wallet log, the order evidence and the PSP verification record. transaction_idPSP record carrying the full mandate-verification breakdown.

13Schema composition & conformance testing

“Compliant” is a claim; these are the proofs. The repo vendors the official UCP schema tree and validates live traffic against it.

How extensions compose

UCP schemas are modular: the base checkout.json stays clean, and each extension ships a $defs["dev.ucp.shopping.checkout"] that augments it via allOf. Per the spec's Resolution Flow, a validator composes the base with every active extension:

composed_checkout = allOf [
  checkout.json,                                          // base capability
  fulfillment.json#/$defs/dev.ucp.shopping.checkout,      // + fulfillment fields
  ap2_mandate.json#/$defs/dev.ucp.shopping.checkout       // + the ap2 object
]

This composition runs both offline (npm run validate) and at runtime: the Shopping Agent's schema resolver loads the vendored schemas, composes the checkout schema for the negotiated capabilities, and validates the checkout response in-flow — per the spec's Resolution Flow (fetch → compose → validate) — surfaced as Schema Resolution / Schema Validation cards in the Protocol Trace.

npm run validate — 20 checks

  • Loads all 87 official schemas into Ajv (draft 2020-12) and resolves every $ref recursively — one broken link fails the run.
  • Audits every ucp.dev URL in our source and inside the live profile documents against the official tree (this catch found a reference to a non-existent catalog.json).
  • Runs the real purchase flow and validates raw wire artifacts: business profiles, platform profile, checkout responses (composed), catalog responses, the order webhook.

npm run e2e — 51 checks

  • Profile structure, Cache-Control, MCP handshake, tools/list, dual-output envelopes.
  • The full happy path with exact totals (minor-unit math), mandate chain, capture, webhook, cross-service state consistency.
  • All the negative attacks from §10.
  • AP2 artifact structure: detached-JWS merchant_authorization, and the SD-JWT+kb checkout_mandate (issuer JWT + disclosures + key-binding JWT with typ=kb+jwt binding aud+nonce).

npm run conformance — 28 checks across 11 categories

The official suite (Universal-Commerce-Protocol/conformance, vendored under conformance-official/) is a Python/pytest harness bound to the REST reference server, so it can't run verbatim against our MCP-transport services. npm run conformance asserts the same categories against our live services and prints a per-category PASS/FAIL report: protocol, binding, checkout_lifecycle, fulfillment, business_logic, idempotency, ap2, card_credential, order, webhook, validation.

Documented simplifications. In-memory keys/state (restart = clean slate); plain HTTP on localhost (signature pipeline unchanged); the checkout and payment mandates are real SD-JWT+kb (issuer + holder key-binding + selective disclosure); biometric approval and the 3-D Secure bank page are simulated; PSP issuer/network authorization, declines and 3DS triggers are simulated by the chosen test card. Both human-present (closed Checkout + Payment Mandates, signed by the user device key) and human-not-present (user-signed open mandates carrying constraints → agent-signed closed mandates) flows are implemented.
